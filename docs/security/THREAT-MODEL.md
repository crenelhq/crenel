# Crenel: Threat Model

> Part of the [third-party audit package](../AUDIT.md). What Crenel defends
> against, and, just as important, what it explicitly does **not**. Read
> [SECURITY-MODEL.md](SECURITY-MODEL.md) first for the guarantees this defends.

---

## 1. What Crenel defends against

Each of these is a failure mode Crenel is specifically built to make
impossible (or, where full prevention isn't possible, to make loud and
declared instead of silent). The vocabulary (`MISREAD-↓`, `MISREAD-↑`,
`MISMANAGE`) is `TOPOLOGY-RISK-REGISTER.md`'s and is worth reusing in findings.

### 1a. Silent fail-open

**The failure:** an edge's default-deny catch-all is missing, absent, or
buried inside config Crenel didn't fully parse, and Crenel reports the wall as
intact anyway.

**The defense:** `DenyCatchAllPresent` is read structurally per-driver, and
the **ENFORCED** verdict is only ever reported when the *entire* config was
parsed (`FullyParsed()`). Any unparsed construct downgrades the claim to
**UNKNOWN**, never a falsely reassuring green. See SECURITY-MODEL.md §1c.

### 1b. Drift (reality diverges from what the operator believes is exposed)

**The failure:** a route or DNS record changes outside of Crenel (hand edit,
another tool, a partial apply) and the operator's mental model, "here's what
I last told Crenel to expose," silently stops matching reality.

**The defense:** there is no mental model to diverge from in the first place.
Crenel has no stored desired state, so every `status`/`audit` call reports
current live truth, and `drift`/`reconcile` explicitly diff live state against
the *canonical exposed set* (itself derived from live, not stored) to surface
or fix exactly this class of divergence.

### 1c. Foreign-route clobbering

**The failure:** Crenel mutates a route that is actually owned/generated by
another tool (Nginx Proxy Manager, a Docker-label proxy, Pangolin, a hand-built
config Crenel didn't create). The mutation read-back-verifies green in the
moment, then is silently reverted the next time the other tool regenerates its
config. This is `TOPOLOGY-RISK-REGISTER.md`'s **#1-ranked** danger
(highest prevalence × highest severity).

**The defense:** the pre-mutation ownership gate (`gateOwnership`) refuses to
touch a route classified `foreign` or `unknown`, for every mutating verb,
before any driver `Apply` runs. See SECURITY-MODEL.md §1e.

### 1d. Whole-zone DNS reconcile deleting real records

**The failure:** a DNS "reconcile to desired state" operation is authoritative
over an entire zone and deletes/mangles records it doesn't understand (MX,
SRV, CAA, a hand-managed wildcard, a second NS delegation) because its
internal model can't faithfully represent them.

**The defense:** two independent guards. (1) The whole-zone `dnscontrol push`
path refuses to push a zone containing any record type/shape it can't
faithfully re-render (multi-field or multi-value records), and separately
refuses to push a zone containing **any** pre-existing record it doesn't
already manage unless the operator opts in with `dedicated_zone: true`, an
explicit assertion of whole-zone ownership. (2) The **surgical** Cloudflare
mode sidesteps whole-zone authority entirely: it is a per-record REST driver
that only ever touches records it created (marker-checked at the primitive
level), so it is safe to point at a live, shared, mixed-content zone. See
`docs/DNS-DESIGN.md` §5, §11.

### 1e. False "protected" claims (auth asserted where none exists)

**The failure:** a public host reports as authenticated/protected when it
actually isn't. This is the worst-direction misread, because it tells the
operator a service is safe when it's on the internet unprotected.

**The defense:** the `public_without_auth` guardrail on the *write* side
refuses to publish a host with auth left unspecified (only an explicit
`--auth <policy>` or `--auth none` proceeds; `--yes` does not bypass it). On
the *read* side, for a front→downstream chain topology, auth is resolved by
**observation** wherever the downstream is readable. A downstream host that
is actually unauthenticated **is flagged**, not blanket-suppressed by the
"auth is asserted downstream" fallback. The fallback assertion is used *only*
when the downstream genuinely cannot be read, and is itself labeled "asserted,
not observed" rather than presented as a confirmed fact.

### 1f. Silently misreporting scope it can't parse

**The failure (general form of 1a/1e):** any config construct Crenel's parser
doesn't fully understand (a nested subroute, a path/header/method matcher, an
indirect backend, a second `http.servers` block) gets silently dropped from
the view instead of surfaced, so `status` under-reports what's actually
exposed.

**The defense:** the `Unparsed[]`/`Coverage()`/ternary-`Ownership` machinery
described in SECURITY-MODEL.md §1c. This is the universal net underneath
1a/1e/1c-elsewhere; almost every entry in `TOPOLOGY-RISK-REGISTER.md`'s §2
register is closed (or bounded) by this single mechanism rather than by
topology-specific modeling.

## 2. What Crenel explicitly is NOT

This section exists so an auditor doesn't spend time evaluating Crenel against
a threat model it never claimed to meet.

- **Not a firewall, WAF, or IDS.** Crenel does not inspect traffic, rate-limit,
  block by IP/signature, or sit in the request path at all. It configures the
  proxy/DNS that do; it has zero runtime presence once a command exits.
- **Not an authentication/authorization system.** Crenel attaches a forward-auth
  policy **by reference**: a name resolved to a per-driver reference (an
  Authelia/Authentik endpoint, a Traefik middleware, an nginx `auth_request`
  location). See `AUTH-DESIGN.md`. It never implements, verifies, or owns the
  auth provider's actual logic, secrets, or session handling. "Crenel says a
  host has policy `authelia` attached" is a claim about *routing configuration*,
  not a claim that Authelia itself is correctly configured or unbypassable.
- **Not a secrets manager.** Credential handling, redaction, and the loopback-
  admin trust model are `SECURITY.md`'s domain, not this doc's. A finding about
  secret leakage belongs against that document's threat model, not this one's
  (though the two compose; see SECURITY-MODEL.md's framing note).
- **Not a sandbox or least-privilege boundary.** Crenel runs with the invoking
  operator's full privileges and whatever SSH/admin access its config grants.
  A compromised operator session compromises the edge; Crenel does not attempt
  to constrain that (`SECURITY.md` §5.3).
- **Not a daemon, and not resilient to being killed mid-apply by design beyond
  `resume`.** Crenel is an on-demand CLI. The `resume` verb re-derives the
  remaining delta from live state after an interrupted apply, but there is no
  background reconciliation loop. If an apply is interrupted and `resume` is
  never run, the edge is left in whatever partial state the last completed
  step produced (declared via the standard error path, not silently).
- **Not a replacement for the edge/DNS software's own security posture.**
  Crenel inherits whatever the underlying Caddy/Traefik/nginx/Cloudflare/
  AdGuard instance is or isn't protected against. If the edge's admin API is
  network-exposed (an anti-pattern `SECURITY.md` calls out explicitly), no
  amount of correctness in Crenel's logic protects the operator. See B3 in
  `SECURITY.md` §4.
- **Not omniscient about topologies it doesn't model.** Config generators
  outside the detected list (§1c above), tunnel/overlay ingress Crenel can't
  read the config of, auth enforced inside the app itself, per-path auth: all
  of these are **declared unknown** rather than silently misreported (the
  detect-and-declare-unknown net). But "declared unknown" is not the same as
  "handled correctly." `KNOWN-LIMITS.md` is the honest list of exactly which
  topologies fall into this bucket today.

## 3. The asymmetry that shapes severity

Two failure directions are not equally dangerous, and findings should be
triaged accordingly (this is `TOPOLOGY-RISK-REGISTER.md`'s framework, restated
for this package):

- **MISREAD-↓ / under-report exposure (or over-report protection).** Reports
  *private* when actually *public*, or *authenticated* when actually *open*.
  **Always the worst class**: it tells the operator a service is safe when it
  is on the internet. Any finding in this class should be treated as
  critical regardless of how narrow the triggering config shape is.
- **MISREAD-↑ / over-report exposure (cry-wolf).** Flags something as public
  or unprotected when it's actually fine. Less dangerous, but real. It erodes
  trust in the tool's warnings and trains operators to ignore them. Worth
  reporting, lower severity.
- **MISMANAGE.** Crenel applies a change, read-back-verifies it **green**, and
  the change is then silently reverted by a force Crenel doesn't model. This
  is uniquely dangerous because it defeats Crenel's own core safety mechanism
  (read-back-verify): the operator has *positive confirmation* the change
  worked, and it's still wrong minutes later. Treat any reproducible MISMANAGE
  as at least as severe as a MISREAD-↓.
